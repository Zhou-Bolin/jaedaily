题目 93/100

在一个多租户 LLM serving 系统里，显存的主要瓶颈是 KV cache。请求里既有短问答，也有长上下文对话；你需要在固定延迟 SLO 下把单卡吞吐提升 30%，同时让回答质量下降不超过可接受阈值。你会如何设计 KV cache 的压缩、淘汰和回退策略？

参考回答

先把问题拆成三层：1) 识别哪些 token 值得保留；2) 用什么表示做压缩；3) 压缩失效时如何回退。

第一步是分层缓存。把最近 token、共享前缀和高注意力权重片段放在热层，剩余内容放在冷层。这样做的原因是，LLM 的注意力并不均匀，很多 token 对后续解码几乎没有贡献。

第二步是设计压缩。可以先做分页式管理，再对冷层做低比特量化或分块摘要；如果模型对某些 head 极其敏感，就保留这些 head 的原始 KV，不对所有维度一刀切。

第三步是在线决策。按请求长度、历史命中率、当前显存水位计算保留预算，用贪心或轻量打分函数做逐层淘汰；复杂度最好控制在 O(n) 或 O(n log n) 以内，避免压缩策略本身成为瓶颈。

第四步是回退。压缩后若 perplexity、logprob 或人工抽检指标明显恶化，立即切回原始 cache；线上可以先对低风险流量灰度，再逐步扩大到长尾请求。

回答分析

强答应覆盖四件事：先把问题定义成资源约束下的在线优化问题，再讲清缓存分层和淘汰准则，然后说明压缩方案不是全局统一，而是要按 head、层、请求类型做差异化，最后说清灰度和回退。

常见错误有三类：一是只讲量化，不讲选择性保留；二是只谈理论，不说在线复杂度和服务指标；三是没有回退方案，默认压缩总是有效。真正的高级候选人会主动补上监控指标、异常回退和 A/B 评估。

Code Challenge 加权缓存淘汰
实现一个容量为 K 的缓存。每个条目有 `weight` 和最近访问时间。收到 `PUT key value weight` 时，若 key 已存在则更新 value/weight 并视为一次访问；若不存在且缓存已满，先淘汰 `weight` 最小的条目，若并列则淘汰最近访问时间最早的条目。`GET key` 若存在返回 value 并更新访问时间，否则返回 -1。`TICK dt` 让当前时间前进 dt。要求所有操作均摊 O(log n) 或更优。

题目 91/100

实现一个容量为 K 的缓存。每个条目有 `weight` 和最近访问时间。收到 `PUT key value weight` 时，若 key 已存在则更新 value/weight 并视为一次访问；若不存在且缓存已满，先淘汰 `weight` 最小的条目，若并列则淘汰最近访问时间最早的条目。`GET key` 若存在返回 value 并更新访问时间，否则返回 -1。`TICK dt` 让当前时间前进 dt。要求所有操作均摊 O(log n) 或更优。

参考解法

把问题拆成两个结构：哈希表负责 O(1) 找到条目，优先队列负责按 `(weight, last_access_time, version)` 找到最该被淘汰的条目。每次访问或更新时不直接删除旧堆节点，而是给条目打版本号，往堆里塞一个新记录，弹出时如果版本不一致就跳过，这样可以避免中间删除的复杂度。

`TICK` 只推进全局时钟，不需要遍历所有元素。`GET` 和 `PUT` 都先更新时间戳，再写入新版本；当容量超限时，循环弹堆顶直到找到一个仍然有效的条目并删除。整体复杂度是均摊 O(log n)，空间会多出少量失效节点，但可通过定期清理控制。

解题分析

高质量答案要说明为什么不能只用普通 LRU：因为淘汰顺序同时依赖权重和最近访问时间，单一链表不够。最好把 `lazy deletion`、版本号、堆顶校验讲清楚，再补上 `GET/PUT/TICK` 的复杂度边界。

常见错误是：直接在线性表里找最小权重，导致 O(n)；或者每次修改都从堆里删除旧节点，复杂度和实现都会变差。面试官还会继续追问并发下怎么做、堆积的失效节点怎么清理，以及如果权重频繁变化是否要换成平衡树。
