算法理论 第一讲

状态空间模型音频建模

▶ YouTube ▶ B站↗ 原始论文
SSM就像一个智能笔记本：平时只需翻到最后一页（当前隐状态）就能继续写，而不必重读所有历史页——但如果你需要，它能在考试前一晚把所有页同时摊开并行复习。
历史演进

音频序列建模的核心矛盾始终是：如何在不牺牲长程依赖捕获能力的前提下，将推理复杂度从二次方降至线性——这一矛盾驱动了从RNN到Transformer再到SSM的整条演进路线。

1990s
经典RNN时代的线性递推

在Transformer出现之前，音频和语音建模的主力是RNN及其变体LSTM（Hochreiter & Schmidhuber, 1997）。RNN本质上是一个离散线性递推系统：$h_t = A h_{t-1} + B x_t$，其推理复杂度为 $O(T)$，内存占用极低。但RNN的致命缺陷是梯度消失/爆炸，难以捕获数千步以上的长程依赖；且无法并行训练，在GPU时代效率极差。

2017
Transformer重塑序列建模范式

Vaswani等人提出的Transformer以全局自注意力机制彻底解决了长程依赖问题，训练可完全并行化。但其代价是推理时的 $O(T^2)$ 时间复杂度和 $O(T^2)$ 的KV Cache内存开销。对于音频这类高采样率序列（24kHz下1秒即24000个时间步），Transformer的二次方代价几乎是灾难性的——这正是音频生成模型长期依赖梅尔谱图等降维中间表示的根本原因。

2021-2022
结构化状态空间模型（S4）的崛起

Albert Gu等人在2021年提出S4（Structured State Space Sequence Model），将连续时间状态空间方程离散化，并通过HiPPO矩阵初始化解决长程记忆问题。S4的核心洞见是：连续SSM $\dot{h}(t) = Ah(t) + Bx(t)$，$y(t) = Ch(t)$ 经离散化后可展开为卷积形式，从而在训练时并行计算（$O(T \log T)$），推理时退化为RNN递推（$O(1)$ per step）。这一"训练像卷积、推理像RNN"的双重身份是SSM的核心价值。

2023
Mamba引入选择性机制

Gu & Dao在2023年提出Mamba，在S4基础上引入输入依赖的选择性（Selective SSM）：矩阵 $B$、$C$、$\Delta$ 不再是固定参数，而是输入 $x_t$ 的函数。这使模型能够动态决定"记住什么、忘记什么"，弥补了S4在内容感知能力上的不足。Mamba同时设计了硬件感知的并行扫描算法（Parallel Selective Scan），充分利用GPU的SRAM层级，实现了比Transformer快5倍以上的推理速度。

2024-2025
Mamba进入音频生成主流

MambaVoiceCloning（本日论文[22]）等工作将门控双向Mamba引入TTS编码器，在推理阶段完全移除注意力层，实现了与Transformer质量相当甚至更优的语音克隆效果，同时推理延迟大幅降低。这标志着SSM从NLP领域向音频生成领域的系统性迁移。

核心思想
状态空间模型（SSM）通过将连续时间动力系统离散化，构建一个可在训练时展开为并行卷积、推理时压缩为常数内存递推的序列建模框架，以线性复杂度捕获任意长程依赖，是Transformer在长序列场景的结构性替代方案。
数学结构

SSM的数学基础来自线性时不变系统理论。连续时间形式为： $$\dot{h}(t) = Ah(t) + Bx(t), \quad y(t) = Ch(t) + Dx(t)$$ 其中 $h(t) \in \mathbb{R}^N$ 为隐状态，$A \in \mathbb{R}^{N \times N}$ 为状态转移矩阵，$B \in \mathbb{R}^{N \times 1}$、$C \in \mathbb{R}^{1 \times N}$ 为投影矩阵，$x(t)$ 为输入标量。 通过零阶保持（ZOH）离散化，步长为 $\Delta$： $$\bar{A} = e^{\Delta A}, \quad \bar{B} = (\Delta A)^{-1}(e^{\Delta A} - I) \cdot \Delta B$$ 离散递推为：$h_t = \bar{A} h_{t-1} + \bar{B} x_t$，$y_t = C h_t$ 关键洞见：将递推展开可得卷积核 $\bar{K} = (C\bar{B},\ C\bar{A}\bar{B},\ \ldots,\ C\bar{A}^{T-1}\bar{B})$，从而 $y = x * \bar{K}$，训练时用FFT计算卷积，复杂度 $O(T \log T)$。 Mamba的选择性扩展使 $B_t = s_B(x_t)$，$C_t = s_C(x_t)$，$\Delta_t = \text{softplus}(s_\Delta(x_t))$，三者均为输入的线性投影。此时卷积核随输入变化，无法预计算，必须用并行前缀扫描（Parallel Prefix Scan）求解，复杂度仍为 $O(T)$，但具备了内容感知能力。 HiPPO矩阵初始化：$A_{nk} = -\sqrt{(2n+1)(2k+1)}$（$n > k$），保证了对历史信号的最优多项式逼近，这是S4能捕获超长程依赖的数学根基。

工作机制

SSM的整体逻辑是：将输入序列通过一个可学习的线性动力系统进行状态压缩，再从状态中读出输出——训练时展开为卷积并行计算，推理时折叠为递推流式处理。

Step 1输入投影与参数生成

输入 $x \in \mathbb{R}^{T \times D}$ 首先经过线性层扩展为内部维度，然后对每个通道独立生成SSM参数。在Mamba中，$B_t$、$C_t$、$\Delta_t$ 通过对 $x_t$ 做线性投影得到，维度分别为 $[B, L, N]$、$[B, L, N]$、$[B, L, D]$。为什么这样设计：固定参数的SSM（如S4）无法区分"重要输入"和"噪声输入"，选择性机制让模型能动态调节状态更新幅度，类似于LSTM的门控但计算更高效。

Step 2离散化（Discretization）

将连续参数 $A$、$B$ 通过步长 $\Delta_t$ 做ZOH离散化，得到 $\bar{A}_t$、$\bar{B}_t$。$\Delta_t$ 越大，表示模型"忽略"当前输入、更多依赖历史状态；$\Delta_t$ 越小，则快速响应当前输入。关键细节：$\Delta$ 经过softplus激活保证正值，且其初始化偏向小值，使模型在训练初期倾向于快速遗忘，避免梯度爆炸。

Step 3并行选择性扫描（Parallel Selective Scan）

核心计算为：$h_t = \bar{A}_t h_{t-1} + \bar{B}_t x_t$。由于 $\bar{A}_t$ 随时间变化，无法用FFT卷积，改用并行前缀扫描（类似GPU上的prefix sum）。算法将长度 $T$ 的序列分块，在SRAM内完成块内扫描，块间通过传递边界状态合并，避免了反复读写HBM。为什么重要：这是Mamba比朴素实现快数倍的工程核心，内存带宽是瓶颈而非计算量。

python # 简化的选择性扫描伪代码 def selective_scan(u, delta, A, B, C):     # u: [B, L, D], delta: [B, L, D], A: [D, N]     deltaA = exp(einsum(delta, A))  # [B, L, D, N]     deltaB_u = einsum(delta, B, u)  # [B, L, D, N]     h = zeros(B, D, N)     ys = []     for i in range(L):         h = deltaA[:, i] * h + deltaB_u[:, i]         y = einsum(h, C[:, i])  # [B, D]         ys.append(y)     return stack(ys, dim=1)
Step 4门控输出融合

Mamba在SSM输出后引入门控机制：$y = \text{SSM}(x) \odot \text{SiLU}(z)$，其中 $z$ 是输入的另一个线性投影。为什么这样设计：纯线性SSM的表达能力有限，门控引入非线性，使模型能学习"何时信任SSM输出"，类似于GRU的更新门，但计算代价极低。

Step 5双向扩展（用于编码器）

在TTS编码器等非因果场景（如MambaVoiceCloning），对序列做前向和后向两次扫描，将两个方向的隐状态拼接或相加。关键细节：双向Mamba的推理复杂度仍为 $O(T)$，而双向Transformer为 $O(T^2)$，在长音频序列上优势显著。

长远价值

SSM/Mamba已在多个工业级系统中得到验证：Apple在设备端语音模型中探索SSM替代Transformer以降低内存占用；多个开源TTS框架（如Kokoro的实验分支）引入Mamba编码器；音频超分辨率、语音增强等实时处理场景因其 $O(1)$ 推理内存而受益。其长远价值在于：只要音频序列长度持续增长（高采样率、长对话、音乐生成），Transformer的二次方代价就始终是瓶颈，SSM提供了理论上唯一兼顾质量与效率的结构性解法。

前沿动向

当前核心开放问题：①Mamba在上下文学习（in-context learning）能力上仍弱于Transformer，机制尚不清楚；②选择性扫描的硬件实现在非NVIDIA平台（如Apple Silicon、TPU）上缺乏优化；③Mamba与Transformer的混合架构（如Jamba）如何最优配比仍是经验性问题；④SSM在多说话人、多轨音频等多流场景的扩展方式尚待系统研究。

工程·思维 第二讲

实时语音增强工程选型

▶ YouTube ▶ B站↗ 原始论文
实时语音增强就像在嘈杂餐厅里做同声传译：你必须在对方说完每句话后立刻翻译，不能等全场安静再开始，所以你需要一套极其高效的"边听边过滤"策略，而不是事后慢慢回放录音来精雕细琢。
历史演进

实时语音增强的工程挑战本质上是一道约束优化题：在严苛的延迟预算（通常<50ms）内，用尽可能少的计算资源，将噪声从语音中分离——这一需求从电话时代就存在，但深度学习的介入彻底改变了解法空间。

1990s-2000s
信号处理时代的统计方法

早期噪声抑制完全依赖信号处理：维纳滤波（Wiener Filter）假设噪声平稳，通过估计信噪比（SNR）计算频域抑制系数；谱减法（Spectral Subtraction）直接从含噪谱中减去估计的噪声谱。这些方法计算量极低（可在DSP芯片上实时运行），但对非平稳噪声（如餐厅、街道）效果很差，且容易产生"音乐噪声"（musical noise）伪影。WebRTC的内置噪声抑制模块（NS/NS1）至今仍基于这一思路，是全球部署量最大的实时降噪方案。

2014-2018
DNN进入语音增强

随着GPU算力普及，基于DNN的语音增强开始出现。早期方案（如Wang et al., 2014的理想二值掩码IBM）在频域预测掩码（mask），将含噪STFT乘以掩码后重建语音。这类方法显著提升了非平稳噪声场景的效果，但推理延迟高（帧级处理+模型前向传播），难以满足实时要求。微软在2015年将DNN降噪集成进Skype，是工业界首个大规模部署案例。

2018-2020
RNNoise与轻量化实时降噪

Mozilla的RNNoise（Valin, 2018）是实时深度降噪的工程里程碑：仅用一个小型GRU网络（<100KB参数），在Opus编解码器的频带特征上预测增益，实现了<1ms的CPU推理延迟。其核心设计哲学是"让信号处理做特征工程，让神经网络做决策"，而非端到端替换整个流程。这一思路影响了后续大量轻量化降噪方案。

2020-2022
DTLN、FullSubNet与流式架构成熟

DTLN（Westhausen & Meyer, 2020）提出双变换层网络，在时域和频域分别建模，参数量仅1M但效果接近重型模型。FullSubNet（ICASSP 2021）引入全频带+子频带联合建模，在DNS Challenge中取得领先。这一时期，流式（streaming）推理架构成为工程标准：模型必须支持帧级输入、维护跨帧状态，不能"看未来"。

2022-至今
基于扩散/Transformer的高质量降噪与实时化挑战

NVIDIA的CleanUNet、微软的DCCRN、以及基于扩散的SGMSE+等模型将降噪质量推向新高度，但推理延迟从数十毫秒到数百毫秒不等，难以直接用于实时场景。工程界的主要工作是通过知识蒸馏、模型剪枝、量化将这些大模型压缩到实时可用的规模，同时保留其质量优势。

核心思想
实时语音增强工程的核心是在延迟、质量、计算资源三者之间做显式权衡：选择合适的处理域（时域/频域）、帧长、模型架构和推理后端，使系统在目标硬件上稳定运行于延迟预算之内，同时达到可接受的降噪质量。
数学结构

实时语音增强的延迟预算分解是工程决策的数学基础。 总端到端延迟由以下部分组成： $$L_{total} = L_{capture} + L_{frame} + L_{lookahead} + L_{inference} + L_{playback}$$ 其中： - $L_{capture}$：麦克风采集缓冲延迟，通常10-20ms - $L_{frame}$：处理帧长，如20ms帧长则贡献20ms - $L_{lookahead}$：模型向前看的帧数 $\times$ 帧移，因果模型此项为0 - $L_{inference}$：模型前向传播时间，必须 $< L_{frame}$（否则实时率<1） - $L_{playback}$：播放缓冲，通常10-20ms 对于50ms总预算，典型分配：$L_{frame}=20\text{ms}$，$L_{inference}<10\text{ms}$，$L_{capture}+L_{playback}=20\text{ms}$。 频域掩码估计的核心公式：含噪信号 $Y = S + N$，在STFT域预测理想比值掩码（IRM）： $$M(t,f) = \frac{|S(t,f)|^2}{|S(t,f)|^2 + |N(t,f)|^2} \in [0,1]$$ 增强语音 $\hat{S}(t,f) = M(t,f) \cdot Y(t,f)$，再经ISTFT重建波形。 实时率（RTF）定义：$\text{RTF} = \frac{T_{inference}}{T_{audio}}$，必须满足 $\text{RTF} < 1$，工程上要求 $\text{RTF} < 0.3$ 以留出余量。

工作机制

实时语音增强系统的整体逻辑是：将连续音频流切分为重叠帧，对每帧独立做特征提取→模型推理→波形重建，同时维护跨帧的模型状态，保证因果性和低延迟。

Step 1流式帧切分与缓冲管理

将输入音频流按固定帧移（hop size，通常10-20ms）切分，每帧与前若干帧拼接形成分析窗口（如20ms帧移+20ms历史=40ms窗口）。为什么不用更长帧：帧长直接贡献延迟，且更长帧意味着更大的STFT频率分辨率但更差的时间分辨率，对快速变化的噪声（如键盘声）响应变慢。关键实现细节：必须使用环形缓冲区（ring buffer）管理输入流，避免内存拷贝；帧切分必须在独立线程中完成，与推理线程解耦，防止抖动（jitter）导致的音频卡顿。

Step 2特征提取（处理域选择）

- 时域直接处理（如SEGAN、DCCRN-E）：直接输入原始波形，无需STFT，延迟最低，但模型需要自己学习频率分解，参数量通常更大。 - 频域掩码预测（主流方案）：对帧做STFT，提取幅度谱/复数谱/对数梅尔谱作为特征，模型预测掩码后重建。工程权衡：STFT的FFT计算在CPU上极快（<0.1ms），频域特征维度低（如257维），模型输入更紧凑，推理更快。 - 子带分解（如RNNoise）：用心理声学滤波器组将信号分解为20-40个子带，每个子带独立处理，计算量极低，适合嵌入式场景。

Step 3模型推理与状态维护

流式推理的核心约束是因果性：模型在时刻 $t$ 只能使用 $\leq t$ 的信息。RNN/GRU天然满足因果性，且隐状态在帧间自动传递；Transformer需要使用因果掩码并缓存KV，内存随上下文长度线性增长。关键工程决策：

# 流式推理状态管理示例 class StreamingEnhancer:     def __init__(self, model):         self.model = model         self.rnn_state = None  # 跨帧持久化         self.input_buffer = RingBuffer(size=frame_size * 3)          def process_frame(self, audio_chunk):         self.input_buffer.push(audio_chunk)         features = extract_features(self.input_buffer.get())         enhanced, self.rnn_state = self.model(             features, self.rnn_state  # 传入并更新状态         )         return reconstruct_waveform(enhanced)

为什么状态管理至关重要：如果每帧重置状态，模型无法利用历史上下文，降噪效果退化为单帧处理，对持续性噪声（如空调声）的抑制能力大幅下降。

Step 4推理后端选择与量化

在目标硬件上选择推理后端： - 浏览器/JS环境：ONNX.js、TensorFlow.js，通常需要INT8量化将模型压缩到<2MB - 移动端（iOS/Android）：CoreML/TFLite，支持Neural Engine加速，可用FP16 - 服务端实时流：TensorRT（NVIDIA GPU）或ONNX Runtime（CPU），批大小=1的推理优化至关重要 - 嵌入式/DSP：需要手写C推理引擎，通常只支持定点运算 关键指标：在目标硬件上测量P99推理延迟（而非平均值），因为实时系统对延迟抖动零容忍。

Step 5重叠相加（OLA）与相位重建

增强后的频域帧经ISTFT重建为时域波形，与前一帧的重叠部分做加权叠加（Overlap-Add）。为什么需要OLA：直接拼接会在帧边界产生不连续性（"咔哒"声）；OLA通过汉宁窗等平滑窗函数保证波形连续性。常见陷阱：如果模型输出的相位与输入相位不一致（如只预测幅度掩码），需要复用输入相位或用Griffin-Lim迭代重建，后者不适合实时场景。

长远价值

实时语音增强是音视频通话（Zoom、Teams、微信）、语音助手（Siri、小爱同学）、直播推流等数十亿用户产品的基础设施。NVIDIA RTX Voice/Broadcast、Apple的麦克风隔离功能、微软Teams的AI降噪均基于这一技术栈。其工程价值在于：这是少数几个"用户能立即感知到质量差异"的AI功能，且部署规模极大，每降低1ms延迟或1%CPU占用都有显著的商业价值。

前沿动向

当前核心挑战：①个性化降噪——针对特定用户的声音特征自适应，而非通用模型；②多说话人场景的目标说话人提取（Target Speaker Extraction）与降噪联合建模；③扩散模型的实时化——SGMSE+等高质量扩散降噪模型推理延迟仍在100ms以上，如何通过一步扩散（consistency model）实现实时化是热点；④端侧神经处理单元（NPU）的专用优化。
